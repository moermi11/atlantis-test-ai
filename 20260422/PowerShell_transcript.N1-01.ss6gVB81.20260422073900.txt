# QEMU ARM64 (`virt`) Smoke Path

## Purpose

This document defines the Sprint 1 smoke path for validating a generic Atlantis ARM64 rootfs under QEMU `virt`.

This path is for generic Atlantis userspace validation only. It does not validate Google Pixel 8 (`shiba`) hardware.

## Status

- documentation: `in progress`
- scripted rootfs input: `in progress`
- recorded smoke result: `unverified`

## Host Prerequisites

Run this path on a Debian-based development host with:

- `qemu-system-arm`
- `qemu-efi-aarch64`
- `qemu-utils`
- `mmdebstrap`
- `dpkg-dev`
- `devscripts`
- `build-essential`

Useful additional tools:

- `virtiofsd` or another host/guest file-sharing path
- `serial console logging` capture through `tee` or `script`

## Artifact Assumptions

This smoke path assumes the following generic artifacts exist:

- Atlantis package `.deb` files built from `packages/`
- a local Atlantis package feed generated by `build/mmdebstrap/build-package-feed.sh`
- a generic ARM64 rootfs directory produced by `build/mmdebstrap/build-rootfs.sh`
- a generic ARM64 kernel and initramfs suitable for QEMU `virt`

Important limitation:

- Sprint 1 does not yet produce the generic ARM64 kernel or initramfs in-repo
- this document therefore defines the validation path and command shape, not a claimed successful boot result

## Generic Boundary

The QEMU path must stay device-agnostic:

- do not include `atlantis-device-google-shiba`
- do not use Pixel 8 boot assets
- do not infer anything about fastboot, AVB, or `shiba` partitions from QEMU results

## Example Validation Flow

### 1. Build Atlantis packages

```sh
cd packages/atlantis-base && dpkg-buildpackage -us -uc -b
cd ../atlantis-shell && dpkg-buildpackage -us -uc -b
cd ../atlantis-branding && dpkg-buildpackage -us -uc -b
cd ../..
```

### 2. Build the Atlantis package feed

```sh
sh ./build/mmdebstrap/build-package-feed.sh
```

### 3. Build the generic ARM64 rootfs

```sh
sh ./build/mmdebstrap/build-rootfs.sh
```

### 4. Pack the rootfs into a disk image for QEMU

One documentation-level example:

```sh
ROOTFS_DIR=out/rootfs/bookworm-arm64
IMAGE=out/qemu/atlantis-arm64-generic.img
mkdir -p out/qemu
truncate -s 4G "$IMAGE"
mkfs.ext4 -F "$IMAGE"
sudo mount -o loop "$IMAGE" /mnt
sudo rsync -aHAX "$ROOTFS_DIR"/ /mnt/
sudo umount /mnt
```

This is an example workflow only. Sprint 1 does not claim this image has been booted successfully.

### 5. Boot the generic image in QEMU

Example command template:

```sh
qemu-system-aarch64 \
    -machine virt,gic-version=3 \
    -cpu cortex-a72 \
    -smp 4 \
    -m 4096 \
    -nographic \
    -serial mon:stdio \
    -kernel /path/to/generic/Image \
    -initrd /path/to/generic/initrd.img \
    -append 'console=ttyAMA0 root=/dev/vda rw' \
    -drive if=virtio,format=raw,file=out/qemu/atlantis-arm64-generic.img \
    -netdev user,id=net0 \
    -device virtio-net-pci,netdev=net0
```

Alternative UEFI-oriented template when a suitable EFI-capable guest layout exists:

```sh
qemu-system-aarch64 \
    -machine virt,gic-version=3 \
    -cpu cortex-a72 \
    -smp 4 \
    -m 4096 \
    -nographic \
    -serial mon:stdio \
    -bios /usr/share/qemu-efi-aarch64/QEMU_EFI.fd \
    -drive if=virtio,format=raw,file=out/qemu/atlantis-arm64-generic.img
```

The kernel, initramfs, root device, and firmware choices remain placeholders until Atlantis records a successful generic ARM64 boot path.

## Smoke Checklist

Treat the path as `smoke-tested` only if evidence shows:

- QEMU started with the intended Atlantis rootfs artifact
- the kernel and initramfs were the generic ARM64 test artifacts, not Pixel 8-specific artifacts
- the guest reached early userspace without an immediate kernel panic
- `systemd` or equivalent early init output appeared on the serial console
- the Atlantis package set was present in the rootfs used for the test
- any failure point was captured precisely if the boot stopped before userspace

## What This Path Can Prove

- the Atlantis generic rootfs can be assembled into a QEMU-usable disk image
- Atlantis package dependencies are installable in a generic ARM64 environment
- generic ARM64 early userspace assumptions can be exercised quickly
- obvious boot-time regressions in package composition or service startup can be caught before hardware testing

## What This Path Cannot Prove

- Pixel 8 bootloader behavior
- `fastboot` flashing safety
- Pixel 8 kernel compatibility
- display, touch, storage, charging, suspend, radio, camera, or modem behavior on real hardware
- whether a `shiba` boot artifact will be accepted by the device

QEMU results do not validate Pixel 8 hardware.

## Evidence Checklist

Record at least:

- date tested
- tester
- host distribution and host kernel
- Atlantis package build identifier or git revision
- rootfs path used
- generic kernel path used
- generic initramfs path used
- exact QEMU command line
- whether the result stayed `unverified`, reached `boots`, or failed as `broken`
- serial console log or transcript path
- short summary of the observed boot stage or failure point
